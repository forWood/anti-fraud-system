## 2. 启动流程

### 2.1 开发环境一键启动

```bash
# 1. 进入项目目录
cd anti-fraud-system

# 2. 编译打包
mvn clean package -DskipTests

# 3. 启动基础设施 (MySQL, Redis, Kafka, ClickHouse)
cd anti-fraud-deployment
docker-compose up -d mysql redis kafka clickhouse

# 4. 等待基础设施就绪 (约60秒)
echo "等待MySQL就绪..."
until docker exec risk-mysql mysqladmin ping -h localhost --silent; do sleep 2; done
echo "MySQL已就绪"

# 5. 初始化数据库
docker exec -i risk-mysql mysql -urisk_user -prisk_pass_2026 risk_db < ../anti-fraud-sql/init-schema.sql

# 6. 启动AI Agent
cd ../anti-fraud-agent
mvn spring-boot:run

# 7. 启动知识库服务 (新终端)
cd ../anti-fraud-knowledge-base
pip install -r requirements.txt
python src/services/knowledge_base_service.py

# 8. 启动交易监控 (新终端)
cd ../anti-fraud-microservices/transaction-monitor
mvn spring-boot:run

# 9. 启动预警管理 (新终端)
cd ../anti-fraud-microservices/alert-management
mvn spring-boot:run

# 10. 启动Kafka交易模拟器
cd ../anti-fraud-data-collection
mvn exec:java \
  -Dexec.mainClass="com.bank.risk.collection.producer.TransactionKafkaProducer" \
  -Dproducer.rate=100 -Dproducer.duration=-1
```

### 2.2 生产环境启动

```bash
# 完整docker-compose启动
cd anti-fraud-deployment
docker-compose -f docker-compose.yml -f docker-compose.prod.yml up -d

# 或 使用K8s部署
kubectl create namespace risk-management
helm install anti-fraud ./kubernetes/helm-chart -n risk-management

# 验证服务状态
kubectl get pods -n risk-management
kubectl get svc -n risk-management

# 查看日志
kubectl logs -f deployment/anti-fraud-agent -n risk-management
```

### 2.3 启动顺序要求

```
启动顺序 (严格):
  1. ZooKeeper
  2. Kafka → MySQL → Redis → ClickHouse
  3. Flink JobManager + TaskManager
  4. AI Agent (等待MySQL/Kafka)
  5. Knowledge Base (等待MySQL)
  6. 微服务 (等待AI Agent)
  7. 数据模拟器 (可选)
```

### 2.4 健康检查

```bash
# 检查所有服务状态
curl http://localhost:8080/actuator/health          # AI Agent
curl http://localhost:8081/actuator/health          # Transaction Monitor
curl http://localhost:8088/health                   # Knowledge Base
curl http://localhost:3000/api/health               # Grafana

# gRPC健康检查
grpcurl -plaintext localhost:9090 \
  com.bank.risk.agent.AntiFraudAgentService/HealthCheck

# Kafka topic检查
kafka-topics --list --bootstrap-server localhost:9092
# 预期输出:
# transaction-raw
# transaction-clean
# transaction-assessed
# alert-created
```

## 4. 监控与告警

### 4.1 Prometheus指标说明

| 指标名称 | 类型 | 说明 |
|----------|------|------|
| `risk_decision_total` | Counter | 决策总数(按decision标签分组) |
| `risk_assessment_duration` | Histogram | 评估耗时分布 |
| `risk_rule_execution_total` | Counter | 规则执行次数 |
| `risk_rule_fired_total` | Counter | 规则触发次数 |
| `knowledge_search_empty_total` | Counter | 知识库空结果次数 |
| `risk_score_distribution` | Summary | 风险评分分布 |

### 4.2 告警规则

| 告警名称 | 触发条件 | 级别 | 处理建议 |
|----------|----------|------|----------|
| HighDecisionLatency | P99 > 500ms (持续5分钟) | Warning | 检查ML服务响应/Flink状态 |
| HighRiskTransactionRatio | 阻断比例 > 30% | Critical | 检查规则是否误触发 |
| RuleEngineExecutionFailure | 规则执行次数=0 | Critical | 重启Agent/检查Drools |
| KnowledgeSearchEmptyRateHigh | 空结果率 > 50% | Warning | 检查FAISS索引/向量模型 |
| ServiceDown | 服务up=0 | Critical | 立即排查/自动重启 |

### 4.3 Grafana监控大盘

**访问地址：** `http://localhost:3000`  
**登录信息：** admin / admin123

**面板概览：**

| 面板 | 说明 |
|------|------|
| 决策延迟 (P50/P95/P99) | 评估服务响应时间 |
| 决策分布 | PASS/REVIEW/BLOCK比例 |
| 决策速率 | 每秒评估请求数 |
| 规则引擎速率 | 规则执行/触发频率 |
| 知识库空结果率 | FAISS检索效果 |

## 5. 故障排查

### 5.1 常见问题与解决

#### 问题1：AI Agent启动失败

**症状：** `java.sql.SQLException: Unable to connect to MySQL`

**排查步骤：**
```bash
# 检查MySQL是否运行
docker ps | grep mysql

# 检查MySQL连接
mysql -h localhost -u risk_user -p -e "SELECT 1"

# 检查application.yml配置
cat anti-fraud-agent/src/main/resources/application.yml | grep MYSQL
```

**解决方案：**
```bash
# 重启MySQL
docker-compose restart mysql

# 等待MySQL就绪后重启Agent
docker-compose restart anti-fraud-agent
```

#### 问题2：决策延迟过高 (P99 > 500ms)

**症状：** Grafana显示P99延迟超过阈值

**排查步骤：**
```bash
# 1. 检查ML服务
curl http://tf-serving:8501/v1/models/fraud_detection

# 2. 检查Flink作业
curl http://flink-jobmanager:8081/jobs

# 3. 检查规则引擎
curl http://localhost:8080/actuator/metrics/risk_rule_execution_duration

# 4. 检查JVM GC
curl http://localhost:8080/actuator/metrics/jvm.gc.pause
```

**解决方案：**
```yaml
# 增加Agent副本数
kubectl scale deployment anti-fraud-agent --replicas=5

# 或调整JVM参数
JAVA_OPTS="-Xms4g -Xmx4g -XX:+UseG1GC -XX:MaxGCPauseMillis=100"
```

#### 问题3：规则不生效

**症状：** 预期触发的规则未触发

**排查步骤：**
```bash
# 1. 检查规则是否启用
mysql -e "SELECT rule_id, rule_name, enabled FROM risk_db.rule_definition"

# 2. 检查DRL文件是否正确加载
docker-compose logs anti-fraud-agent | grep "Loading rule"

# 3. 测试单条规则
curl -X POST http://localhost:8080/api/v1/rules/test \
  -H "Content-Type: application/json" \
  -d '{"rule_id": "R001", "features": {"amount": 800000}}'
```

**解决方案：**
```bash
# 强制重新加载规则
curl -X POST http://localhost:8080/api/v1/rules/reload
```

#### 问题4：知识库检索返回空结果

**症状：** `knowledge_search_empty` 指标持续增长

**排查步骤：**
```bash
# 1. 检查FAISS索引
python -c "
from faiss_index_manager import FAISSIndexManager
idx = FAISSIndexManager(768, './data/faiss_index')
print(f'Index size: {idx.get_total()}')
"

# 2. 检查MySQL案例表
mysql -e "SELECT COUNT(*) FROM risk_db.fraud_case"

# 3. 检查向量模型
python -c "from sentence_transformers import SentenceTransformer; m=SentenceTransformer('shibing624/text2vec-base-chinese'); print('OK')"
```

**解决方案：**
```bash
# 重建FAISS索引
python src/services/knowledge_base_service.py --rebuild-index
```

#### 问题5：Kafka消息堆积

**症状：** `kafka_consumer_lag` 持续增大

**排查步骤：**
```bash
# 检查消费组lag
kafka-consumer-groups --bootstrap-server localhost:9092 \
  --group fraud-detection-group --describe

# 输出:
# GROUP                   TOPIC           PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
# fraud-detection-group   transaction-raw 0          10234           55678           45444 ⚡
```

**解决方案：**
```bash
# 增加Flink并行度
# 修改flink.parallelism: 4 → 8

# 或增加Kafka分区数
kafka-topics --alter --topic transaction-raw \
  --partitions 8 --bootstrap-server localhost:9092
```

## 6. 备份与恢复

### 6.1 备份策略

| 数据类型 | 备份方式 | 频率 | 保留期限 |
|----------|----------|------|----------|
| MySQL数据 | mysqldump | 每日凌晨3:00 | 30天 |
| Redis数据 | RDB快照 | 每小时 | 7天 |
| ClickHouse | ALTER FREEZE | 每周日 | 90天 |
| 规则文件 | Git仓库 | 实时 | 永久 |
| FAISS索引 | 文件备份 | 每次重建后 | 30天 |

### 6.2 备份脚本

```bash
#!/bin/bash
# auto-backup.sh - 每日自动备份

BACKUP_DIR="/backup/anti-fraud/$(date +%Y%m%d)"
mkdir -p $BACKUP_DIR

# 1. 备份MySQL
mysqldump -h localhost -u risk_user -p'risk_pass_2026' \
  --single-transaction --routines --triggers \
  risk_db > $BACKUP_DIR/risk_db.sql

# 2. 备份规则文件
cp -r anti-fraud-agent/src/main/resources/rules/ $BACKUP_DIR/rules/

# 3. 备份FAISS索引
cp anti-fraud-knowledge-base/data/faiss_index/* $BACKUP_DIR/faiss/

# 4. 压缩
tar -czf $BACKUP_DIR.tar.gz $BACKUP_DIR
rm -rf $BACKUP_DIR
```

### 6.3 恢复流程

```bash
# MySQL恢复
mysql -h localhost -u risk_user -p risk_db < /backup/anti-fraud/20260514/risk_db.sql

# Redis恢复
docker-compose stop redis
cp /backup/redis/dump.rdb /data/redis/
docker-compose start redis

# FAISS索引恢复
cp /backup/anti-fraud/20260514/faiss/* anti-fraud-knowledge-base/data/faiss_index/

# 验证恢复
curl http://localhost:8080/actuator/health
curl http://localhost:8088/health
```

## 7. 紧急操作

### 7.1 紧急止损

```bash
# 场景: 发现大规模欺诈攻击, 需立即提高风控等级

# 1. 降低风险阈值（高风险从80降到60）
mysql -e "UPDATE system_config SET config_value='60' WHERE config_key='high_risk_threshold'"

# 2. 增加阻断规则权重
# 编辑 fraud-rules.drl，临时提高权重
vi rules/fraud-rules.drl

# 3. 强制重新加载规则
curl -X POST http://localhost:8080/api/v1/rules/reload

# 4. 临时关闭某个渠道
mysql -e "UPDATE channel_config SET enabled=0 WHERE channel_code='POS'"
```

### 7.2 服务降级

```bash
# 场景: ML服务故障, 需要降级到纯规则模式

# 1. 设置ML权重为0
mysql -e "UPDATE system_config SET config_value='0.0' WHERE config_key='ml_weight'"

# 2. 设置规则权重为1.0
mysql -e "UPDATE system_config SET config_value='1.0' WHERE config_key='rule_weight'"

# 3. 刷新Agent配置
curl -X POST http://localhost:8080/api/v1/config/reload
```

### 7.3 系统开关

```sql
-- 全局服务开关 (紧急情况下快速关闭)
UPDATE system_config SET config_value = 'OFF' WHERE config_key = 'global_service_status';

-- 交易监控开关
UPDATE system_config SET config_value = 'OFF' WHERE config_key = 'transaction_monitor_enabled';

-- 预警推送开关
UPDATE system_config SET config_value = 'OFF' WHERE config_key = 'alert_notification_enabled';
```
